# Architecture

The recommended shape is intentionally small:

```text
controller (agent / CI / script)
  -> command envelope
  -> policy gate
  -> Python observer/controller
  -> pyrcl
  -> TRACE32 Remote API on port 20000
  -> TRACE32 PowerView
  -> probe and target
```

The controller should not talk directly in free-form TRACE32 commands. It should
submit a command envelope with intent, risk, timeout, expected state, and
evidence requests. The observer/controller is responsible for translating that
intent into pyrcl calls and for collecting status after each action.

## Control Plane

The control plane sends commands:

- `dbg.cmd(...)` for single TRACE32 commands;
- `dbg.fnc(...)` for TRACE32 functions such as `STATE.RUN()`;
- `dbg.cmm(path, timeout=...)` for CMM scripts that should block with a
  bounded timeout;
- non-blocking `DO` only when a separate observation loop owns completion.

## Observation Plane

The observation plane samples state:

- connection health with `ping()`;
- TRACE32 messages with `get_message()`;
- PRACTICE state with `T32_GetPracticeState` semantics where available;
- debugger state with `get_state()`;
- target run state with `STATE.RUN()`;
- selected window or AREA text with window-content APIs where available;
- output files written by approved CMM scripts.

Observation is not optional. It is how the controller distinguishes command
acceptance, script completion, target stop, hidden dialog, communication
failure, and real test success.

## Policy Plane

The policy plane classifies commands:

- `observe`: reads state, messages, symbols, variables, registers, windows;
- `control`: changes execution state, for example bounded `Go` or `Break`;
- `mutate`: changes target memory, registers, flash, clocks, reset state, or
  watchdogs;
- `danger`: can brick, erase, unlock, leak firmware, or hide safety failures.

Automated controllers may run `observe` and selected `control` commands after
validation. `mutate` and `danger` commands require a local board-approved runbook.

## Evidence Plane

Every non-trivial action should produce evidence:

- JSON action result;
- TRACE32 message text and type;
- PRACTICE state before and after;
- target state before and after;
- relevant AREA/window lines;
- path to logs generated by CMM scripts;
- timeout or recovery classification.

This turns "the agent tried something" into a debuggable audit trail.
